Build 88 focuses on authentication reliability, room browsing, and audio/session stability across VoiceLink clients.

What changed:
- Supported authentication flows should be available consistently across platforms, including Client Account, VoiceLink account, and server-provided browser sign-in methods.
- Successful Mastodon or other browser authentication should return to VoiceLink instead of leaving users on the website.
- Microphone permission prompts should appear only when needed and should not repeat after permission is granted.
- Room and server browsing should keep users focused on the selected server instead of disconnecting or kicking them out after previewing a room.
- The macOS server browser uses real tabs for Servers and Federation so the active browsing context is clearer.
- Room messages, system messages, visible users, and admin room actions should be easier to find and verify in supported rooms.
- Audio join behavior continues to be tuned for lower buffering and clearer microphone status reporting.

Please test:
- Sign in with Mastodon, Client Account, or another supported server method and confirm VoiceLink returns to the app after authentication.
- Join the Community server, browse rooms, preview a room, and confirm the selected server remains active.
- In a room, confirm users, room messages, system messages, and available admin actions appear where expected for the account role.
- Confirm microphone permission is not requested repeatedly after it has already been granted.
- Confirm room audio starts without repeated system-message chatter and report any buffering, clipping, or delayed microphone status.
