VoiceLink Admin Guide

Admin features are role-based. Only authorized roles can edit server settings.

Desktop Admin Access

  • Desktop app refreshes admin role on login and server reconnect.
  • Server Administration menu appears only when admin role is active.

Federation and Default Peer Control

  • Default peer servers such as the main VoiceLink host and the community VPS are managed by the server API and should only be toggled by authorized admins.
  • Members can filter what they see locally, but they should not be allowed to rewrite default server federation policy from client preferences.
  • Use trusted servers for permanent peers and blocked servers for installs that must never exchange room or room-state data with this server.

Maintenance Handoff

  • Enable Maintenance Mode before restart, migration, or emergency repair when you want to keep users from being dropped.
  • Auto-handoff should target a trusted, online VoiceLink peer that already shares synchronized room data with this server.
  • Use the main server or community VPS as the handoff target only when both sides are healthy and federation is already configured.
  • Set the recommended client prompt mode to control whether users are prompted every time, only once, automatically follow a saved action, or follow the server recommendation by default.
  • A user’s explicit client-side choice always overrides the server recommendation. The server recommendation is the default, not a forced policy.
  • When the destination room already exists, VoiceLink should reuse that room instead of creating a duplicate. Users are moved into the existing room.
  • If the destination room does not have enough capacity for incoming users, VoiceLink temporarily expands the room limit so the transfer completes without dropping participants.
  • When capacity is auto-expanded during a transfer, admins should receive a notification and can later keep or revert the raised limit after the room empties.
  • Prepare the target server before taking the source server down. The handoff should leave the target room ready to accept reconnecting clients immediately.

Moderation Controls

  • User moderation: kick, ban, mute policy, room access enforcement.
  • Room moderation: lock, privacy behavior, and preview restrictions.

Messaging and Retention

  • Direct messages replace the room chat panel while a conversation is open. Users return to room chat with the Back action.
  • Initial room and direct-message history should load the latest 20 messages by default unless the admin changes the initial load count.
  • Scrollback should stay capped by the configured retention window and the configured message cap, with an absolute hard limit of 5000 messages.
  • Delete room messages when empty only when the room policy allows it. Attachment-linked messages should remain available until the attachment expires or is removed.
  • Guest retention, authenticated-user retention, and bot memory retention should be set explicitly in Server Config so moderation and storage behavior stay predictable.
  • Use Skip to Latest when reviewing older history so moderators can return to the current message stream without leaving the room.

Room Bots and Assistant Behavior

  • Bots should publish a short status message in the room so users know which commands or mentions they accept.
  • Non-audio bots should not expose mute, solo, or user-volume controls. Show those controls only for bots that actually produce room audio such as TTS or broadcast sources.
  • Bot memory retention should be configured separately from room chat retention when the bot needs longer per-user context.
  • Use /bot help and direct mentions such as @VoiceLink Bot for room-level bot actions until richer per-bot command routing is enabled.

Audio and Policy

  • Publish baseline settings for noise suppression, echo cancellation, and gain control.
  • Require room-only sounds for join/leave and avoid autoplay UI noise on page load.

Update Operations

  • Keep macOS package filename stable as VoiceLinkMacOS.zip.
  • Refresh checksum on every new build, even when version remains 1.0.
  • Fallback to backend IP endpoints only when domain resolution fails, then auto-return to domain-based endpoints when healthy.

Deployment Manager Module

  • Deployment Manager is a built-in module intended for server owners and authorized admins only.
  • Use it to generate a fresh VoiceLink deployment bundle, upload it by SFTP, SMB, HTTP, or HTTPS, and bootstrap an existing remote VoiceLink install through its API when available.
  • Fresh install bundles should be uploaded to the target server and extracted there so deploy.json can be loaded by the new install.
  • Existing install updates can use the remote API bootstrap path so server config, federation peers, and API defaults stay aligned with the main server.
  • If the target install exposes a trusted restart endpoint, the deployment manager can trigger a post-bootstrap restart so the new config is applied without a manual shell login.
  • Owner notification email should include upload destination, API bootstrap target, and getting-started instructions for the receiving server owner.

Log Collection

  • Enable Send Logs in Advanced settings for support diagnostics.
  • Collect app version, endpoint used, and room join/audio errors.
